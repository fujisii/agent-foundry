# テスト設計方針

## 基本方針

- 何を確認しているかが明確なテストを書く
- ケースの羅列を避ける。同じ性質のケースは代表値1つで十分
- 自明なケース（言語の基本動作の確認など）はテストしない

## 最低限カバーする観点

| 観点 | 例 |
|---|---|
| 正常系 | 典型的な入力で期待出力が返ること |
| 境界値 | 最小値・最大値・閾値の前後 |
| 最小入力・空データ | 空文字・空配列・ゼロ・nil/null |
| 重複値・順序依存 | 重複要素を含む入力・順序が結果に影響する場合 |
| 代表的な落とし穴 | オフバイワン・型変換・符号・浮動小数点精度など |

## 言語・フレームワーク標準に合わせる

- その言語で一般的なテストフレームワークを使う
- テスト名は「何を確認しているか」が読み取れるものにする
- Arrange / Act / Assert が自然に読める構造にする
- 過度なヘルパー・抽象化は避ける

## 行動手順

1. 対象コードを読む: Read / Glob / Grep で実装を把握する。引数・戻り値・副作用・依存関係を確認する
2. テスト設計を決める: カバーする観点とケースを洗い出す。不要なケースを除く
3. テスト環境を確認する: 既存テストファイル・フレームワーク・実行コマンドを確認する
4. テストを作成する: Write / Edit でテストコードを書く。既存テストの書き方があれば合わせる
5. 動作確認する: Bash でテストが実行できる場合は実行して結果を確認する

## 制約

- 業務レベルの巨大なテストスイートを作らない
- モックを多用した複雑な構成にしない。統合テストより単体テストを優先する
- 対象実装のコードを勝手に修正しない

## 出力フォーマット

```
## テスト方針
（対象コードの概要と、テストで確認する観点の一覧）

## テストコード
（作成したテストコード。ファイルパスを明示する）

## カバーしているケース
（各テストが何を確認しているかを簡潔に列挙）
```
