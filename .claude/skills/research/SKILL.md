# research

技術調査・情報収集を構造的に行うためのスキル。調査プロセスを6フェーズに分割し、ソースの信頼性を評価しながら、再現性のある出力を生成する。特定の技術分野に依存せず、あらゆる技術トピックに適用できる。

## 調査プロセス

### Phase 1: 問題定義

調査を開始する前に、以下を明確化する。

- **調査対象**: 何を知りたいのか（概念・仕様・比較・原因分析など）
- **スコープ**: 調査範囲の境界（バージョン・言語・ユースケースの限定など）
- **前提条件**: 既知の情報・前提として扱う事柄
- **除外事項**: 調査しないこと、またはその理由

引数でトピックが与えられた場合はそれを出発点とし、曖昧な場合は「このトピックで何を知りたいか」を一文で定義してから調査を開始する。

### Phase 2: キーワード設計

効果的な検索クエリを設計する。

- 主キーワードを抽出し、英語対訳を用意する（技術文書は英語が一次ソースになることが多いため）
- 関連語・同義語・上位概念・下位概念を列挙する
- 否定語（"not", "avoid", "alternative"）を使って不要な結果を除外するクエリも用意する
- 検索クエリは「公式ドキュメント向け」「論文向け」「実装向け」に分けて設計する

### Phase 3: 一次ソース探索

以下の優先順位でソースを探索する。

1. **公式ドキュメント** — 仕様書・言語/フレームワークのリファレンス・RFC・規格文書
2. **大学資料・教科書** — 講義ノート・教科書・技術書（査読あり）
3. **論文** — arXiv・ACM Digital Library・IEEE Xplore・Google Scholar
4. **技術書・書籍** — O'Reilly・Manning など信頼性の高い出版社
5. **信頼できる技術ブログ** — 著者が明記され、根拠のある記事

各ソースは `WebSearch` と `WebFetch` を使って取得する。一次ソースが見つからない場合は「準一次ソース」と明記する。

**Wikipedia の扱い**: 概念の初期把握や関連語の発見には使用してよいが、最終的な根拠としては使わない。使用した場合は「準一次ソース（Wikipedia）」と明示し、より信頼性の高いソースで検証を試みる。

### Phase 4: 信頼性評価

取得した情報について以下を評価する。

| 評価軸 | 確認内容 |
|--------|----------|
| 著者・組織 | 専門家か、当該技術の開発元か |
| 公開日 | 情報が現在も有効か（技術の変化速度を考慮） |
| 査読・レビュー | 論文なら査読有無、ドキュメントなら公式か否か |
| 一次性 | 原典か、他ソースの引用・転載か |
| 根拠の明示 | 主張に対してデータ・実験・仕様が示されているか |

信頼性が低いと判断したソースは使用せず、代替ソースを探す。代替が見つからない場合は不確実性として明示する。

### Phase 5: 情報統合・要約

収集した情報を以下の観点で統合する。

- 複数ソースで一致している事項を「確認済みの事実」として扱う
- ソース間で矛盾がある場合は両論を並記し、信頼性の高い方を優先しつつ差異を明示する
- 原文が英語の場合は和訳を提供し、和訳と解釈を分けて記述する
- 定量的なデータ（計算量・パフォーマンス・バージョン番号など）は正確に転記する

### Phase 6: 不明点・追加調査事項の整理

調査完了後、以下を整理する。

- 調査中に解消できなかった疑問点
- 信頼できるソースが見つからなかった事項
- 調査スコープ外だが関連する重要なトピック
- 今後の調査で検証すべき仮説

---

## 禁止事項

以下は行わない。

- **根拠のない断定**: ソースなしで「〜である」と断言する
- **出典なしの知識列挙**: 学習データからの知識をそのまま事実として提示する
- **Wikipedia の無条件利用**: 一次ソースとして扱う
- **古い情報の無検証利用**: 公開日を確認せずに引用する
- **解釈と事実の混同**: 「〜と述べられている」（事実）と「〜と考えられる」（解釈）を区別しない

---

## 出力フォーマット

調査結果は必ず以下の構造で出力する。セクションは省略しない。情報がない場合は「該当なし」または「確認できなかった」と明記する。

```
## 結論

[1〜3文で端的に答える。断定できない場合は「〜と考えられる」「〜の可能性がある」などの表現を使う]

## 根拠（出典付き）

[各根拠を箇条書きで列挙。形式: 「内容 — 出典名（URL または書籍名・著者・年）」]

- 〜である — [出典名](URL)
- 〜が確認されている — 著者名, 書籍名, 年, p.XX

## 和訳（原文が英語の場合）

[引用した英語原文の和訳。複数ある場合はソースごとに分ける。日本語ソースのみの場合は省略]

## 解釈

[事実に基づいた自分の解釈・考察。「〜と解釈できる」「〜の背景には〜があると思われる」などの形で記述]

## 不確実性

[信頼できるソースが見つからなかった点、ソース間の矛盾、情報の鮮度に問題がある点など]

## 追加調査が必要な点

[今回の調査で解消できなかった疑問、スコープ外だが重要なトピック、検証すべき仮説]
```
