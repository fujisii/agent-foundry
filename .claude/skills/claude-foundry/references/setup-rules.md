# Rules セットアップガイド

## Rules とは

CLAUDE.mdをパス条件付きでモジュール化したもの。`.claude/rules/`ディレクトリに置くMarkdownファイルで、YAMLフロントマターで`globs`を指定すると**該当ファイルを操作するときだけ**ロードされる。

## CLAUDE.md との違い

| | CLAUDE.md | Rules |
|---|---|---|
| ロードタイミング | 常時 | 該当ファイル操作時のみ |
| コンテキスト消費 | 常に消費 | 必要な時だけ消費 |
| 向き不向き | 全般的な指示 | 特定ファイル種別のルール |

## ファイル配置

```
.claude/rules/          ← プロジェクト（git管理・チーム共有）
├── testing.md
├── api-conventions.md
└── frontend/
    └── react-patterns.md

~/.claude/rules/        ← グローバル（全プロジェクト共通）
└── my-coding-style.md
```

## ファイルフォーマット

```markdown
---
globs:
  - "src/**/*.test.ts"
  - "**/*.spec.ts"
alwaysApply: false
---

# テスト規約

- テスト名は日本語で記述する
- `describe` → `it` の2層構造を使う
- モックは最小限に留め、統合テストを優先する
```

### フロントマターフィールド

| フィールド | 説明 |
|---|---|
| `globs` | 適用するファイルパターン（省略可） |
| `alwaysApply` | `true`にするとCLAUDE.mdと同様に常時適用 |

`globs`を省略すると常時適用（CLAUDE.mdに書く代替として使える）。

## よく使うglobパターン

```yaml
# TypeScriptファイル全般
globs: ["**/*.ts", "**/*.tsx"]

# テストファイルのみ
globs: ["**/*.test.ts", "**/*.spec.ts", "**/__tests__/**"]

# APIディレクトリ
globs: ["src/api/**"]

# 複数の拡張子
globs: ["src/**/*.{ts,tsx,js,jsx}"]

# 設定ファイル
globs: ["*.config.*", ".eslintrc*"]
```

## 確認すべき質問

1. どのファイル種別に適用しますか？（例: `*.ts`, `src/api/**`, テストファイルのみ）
2. 常時適用（`alwaysApply: true`）ですか？それとも該当ファイル操作時のみですか？
3. どんなルールを定義しますか？（コーディング規約・設計パターン・禁止事項等）

## ルールの書き方のコツ

- **具体的かつ検証可能**に書く（「良いコードを書く」ではなく「関数は20行以内にする」）
- コンテキストが必要なら例を示す
- 禁止事項は理由も添える（なぜそのルールがあるかをClaudeが理解できるように）

## よくあるユースケース別テンプレート

### テスト規約（testing.md）
```markdown
---
globs: ["**/*.test.*", "**/*.spec.*"]
---
# テスト規約
- テストは振る舞いを検証する（実装の詳細ではなく）
- `beforeEach`でテストデータをリセットする
- データベースのモックは禁止（統合テストを使う）
```

### API規約（api-conventions.md）
```markdown
---
globs: ["src/api/**"]
---
# API規約
- すべてのエンドポイントは入力バリデーションを含む
- エラーレスポンスは `{ error: string, code: string }` 形式
- 認証不要のエンドポイントには必ずコメントでその理由を書く
```

## 作成後のアクション

- 対象ファイルを開いてClaudeに質問すると、ルールが適用されていることを確認できる
- `alwaysApply: false`の場合、対象外ファイルではルールはロードされない
